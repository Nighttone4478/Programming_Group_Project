# 摘要報告

## 專案簡介

本專案以 C 語言實作線上暗棋對戰 AI，目標是讓程式能夠連接指定的暗棋伺服器房間，接收伺服器傳回的棋盤狀態，並根據目前局面自動判斷下一步行動。程式透過 `dark_chess_client.h` 提供的連線函式完成自動加入房間、接收棋盤更新與送出行動指令。

## 功能說明

- 使用 `dark_chess_client.h` 連接暗棋伺服器
- 自動加入線上對戰房間
- 接收伺服器傳回的 JSON 棋盤資料
- 解析棋盤中的 `Covered`、`Null` 與各種棋子名稱
- 判斷目前玩家角色與棋子顏色
- 依照暗棋規則產生合法走法
- 支援翻棋、移動與吃子
- 支援炮隔一子吃子的規則判斷
- 支援兵卒吃將帥、將帥不能吃兵卒等特殊規則
- 使用搜尋式 AI 自動決策
- 送出行動前進行合法性檢查
- 加入出手延遲，避免動作過快造成棋盤同步錯誤
- 可持續進行線上對戰直到遊戲結束或連線中斷

## 執行方式

請先確認已具備 C 語言編譯環境，並將 `ai.c` 與 `dark_chess_client.h` 放在同一專案目錄中。

GCC 環境：

```bash
gcc ai.c -o ai.exe -lws2_32 -lm
```

編譯完成後執行：

```bash
ai.exe
```

程式會自動連線、加入房間，並等待伺服器傳回棋盤狀態。輪到本方行動時，AI 會自動計算並送出操作。

## 程式架構說明

### 1. 線上連線流程

程式啟動後，會先呼叫連線相關函式與伺服器建立連線，接著自動加入房間。成功加入後，程式會持續接收伺服器傳回的棋盤資料，並判斷目前是否輪到本方行動。

主要流程如下：

```text
初始化連線
→ 自動加入房間
→ 接收棋盤 UPDATE
→ 判斷目前狀態是否為 playing
→ 判斷是否輪到本方
→ AI 計算行動
→ 送出 move / flip 指令
```

### 2. 棋盤資料解析

伺服器傳回的棋盤資料包含 32 格暗棋狀態。程式會將每一格轉換成內部棋盤格式：

```text
Covered：尚未翻開的棋
Null：空格
Red_King / Black_King 等：已翻開棋子
```

程式使用一維陣列表示 4×8 棋盤，使搜尋與合法走法判斷更方便。

### 3. 暗棋規則判斷

程式支援以下規則：

- 未翻開棋子只能翻開，不能移動或被吃
- 一般棋子只能上下左右移動一格
- 一般棋子依階級判斷是否可以吃子
- 兵卒可以吃將帥
- 將帥不能吃兵卒
- 炮可以隔一個棋子吃同列或同行的敵方棋
- 不允許移動到 Covered 格
- 送出前會再次檢查指令是否合法

### 4. AI 決策邏輯

AI 的決策流程以「吃子優先」為主要設計方向：

```text
如果有可吃棋子
→ 優先搜尋最佳吃子行動

如果沒有可吃棋子
→ 搜尋一般移動

如果沒有合適移動且仍有暗棋
→ 評估是否翻棋

如果棋盤完全翻開
→ 只使用明棋搜尋，不再進行翻棋計算
```

此設計可以避免 AI 在明明能吃子的情況下仍花大量時間計算翻棋，也能避免全明棋殘局時因翻棋評估造成卡住。

### 5. 殘局強化

在殘局或接近殘局時，AI 會額外考慮位置因素：

- 炮是否能形成炮架
- 炮是否能利用炮架攻擊敵方棋子
- 大子是否能靠近並壓迫敵方棋子
- 敵方棋子是否被限制逃跑路線
- 吃子後是否能形成更好的包圍局面

殘局判斷條件為：

```c
count_covered(b) == 0 ||
(count_covered(b) <= 4 && count_visible_live_pieces(b) <= 8)
```

也就是只有在全部翻開，或暗棋剩很少且明棋數量也少時，才啟用殘局位置評估，避免中前期受到過度干擾。

## 參考論文與演算法依據

### 一、參考論文

本次 AI 架構參考以下論文：

> 謝曜安（2008）。《電腦暗棋之設計及實作》（The Design and Implementation of Computer Dark Chess）。國立臺灣師範大學資訊工程研究所碩士論文。

該論文指出，暗棋屬於不完全資訊且含機率性的棋類遊戲，若直接使用一般遊戲樹搜尋，同時展開「走棋」與「翻棋」會造成分支數過大，難以深入搜尋。因此本專案在設計 AI 時，沒有將所有翻棋可能完整展開，而是採用「走棋與翻棋分開決策」的方式。

### 二、資料結構參考

論文中提出以一維陣列表示 4×8 棋盤，並搭配棋盤—棋子映射結構以提升搜尋效率。本專案參考此概念，使用一維棋盤索引 `0 ~ 31` 對應 4×8 棋盤，並建立棋種位置表以加速合法走法產生。

本專案也參考論文中的預處理概念，建立：

- `MoveTab`：預先儲存每一格上下左右可到達的位置
- `OrderTab`：預先儲存棋子之間是否可以吃子的關係

此設計可以減少搜尋時重複判斷方向與吃子階級的成本。

### 三、搜尋演算法參考

本專案 AI 的主要搜尋核心參考論文中的搜尋架構，包含：

- NegaMax 搜尋
- Alpha-Beta 剪枝
- Quiescence Search（寧靜搜尋）
- Transposition Table
- Zobrist Hash
- 循環剪裁
- 最佳路徑輸出
- 走棋與翻棋抉擇

其中 NegaMax 是 Minimax 在零和遊戲中的簡化寫法，搭配 Alpha-Beta 剪枝可以減少不必要的節點展開。當搜尋深度到達限制時，程式不會直接停止，而是進入 Quiescence Search，只延伸吃子相關局面，以降低水平效應造成的誤判。

### 四、走棋與翻棋抉擇

論文中提到暗棋困難點在於「明棋走法」與「暗棋翻牌」同時存在，若全部展開會造成分支爆炸。本專案依照此方向，將 AI 決策拆成兩部分：

```text
先搜尋最佳走棋
再在必要時評估翻棋
最後比較兩者是否值得翻棋
```

此外，為符合實戰需求，本專案加入「吃子優先」策略：

```text
若目前有可吃棋子
→ 直接進入吃子搜尋
→ 不再評估翻棋
```

這可以避免 AI 在明顯有吃子機會時仍花時間計算翻棋。

### 五、與原論文不同的調整

本專案並非完整照搬論文實驗環境，而是根據線上對戰平台需求做出以下調整：

- 加入 `TIME_LIMIT_MS`，避免搜尋時間過長導致程式卡住
- 加入 `ACTION_DELAY_MS`，避免 AI 送出動作過快造成棋盤同步問題
- 若有可吃棋子，優先處理吃子，不進入翻棋評估
- 全明棋時直接使用明棋搜尋，不再進行翻棋判斷
- 殘局位置評估只在真正殘局或接近殘局時啟用
- 送出指令前再次檢查是否合法，避免伺服器判定違規

這些調整主要是為了配合線上平台即時對戰、非法指令直接判輸與伺服器同步等限制。

## 遇到的問題與解決方式

### 問題一：AI 在有棋可吃時仍然計算翻棋

早期版本中，AI 會在 Covered 較少時花大量時間評估翻棋，導致即使有明顯吃子機會，也可能延遲或選擇翻棋。

解決方式：

```text
根節點先檢查是否有吃子
如果有吃子，直接進入吃子搜尋
不再評估翻棋
```

### 問題二：棋盤完全翻開後容易卡住

當棋盤沒有 Covered 時，理論上不應再進行翻棋判斷。早期版本仍可能進入不必要的翻棋相關計算，造成運算時間過長。

解決方式：

```text
covered == 0 時直接使用明棋 NegaMax 搜尋
完全跳過翻棋評估
```

### 問題三：AI 動作太快導致棋盤同步異常

線上對戰時，如果 AI 太快送出指令，可能造成前端畫面或伺服器狀態尚未同步完成。

解決方式：

```c
#define ACTION_DELAY_MS 1000
Sleep(ACTION_DELAY_MS);
send_action(action);
```

讓 AI 在送出動作前等待一小段時間，提高線上對戰穩定性。

### 問題四：殘局時 AI 不會主動利用炮或包圍

早期 AI 主要依靠立即吃子與材料分數，較少主動製造下一步的吃子機會。

解決方式：

```text
加入殘局位置評估：
炮架分數
包圍分數
限制敵方逃路分數
```

使 AI 在殘局時更容易利用炮線、較大棋子與位置壓迫來取得優勢。
